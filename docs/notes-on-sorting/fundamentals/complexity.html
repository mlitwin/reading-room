Complexity

Matthew Litwin 2026-05-17

Sorting nnn items by comparison can’t be faster than O(nlog⁡n)O(n \log n)O(nlogn). The lower bound comes from information theory: there are n!n!n! possible orderings, each comparison gives one bit of information, so any decision tree resolving the ordering has height at least log⁡2(n!)=Θ(nlog⁡n)\log_2(n!) = \Theta(n \log n)log2​(n!)=Θ(nlogn).

Below that bound live the non-comparison sorts — counting sort and radix sort — which exploit structure in the keys to beat nlog⁡nn \log nnlogn for specific inputs. They’re a different beast.

For comparison sorts, the spectrum is:

  • O(n2)O(n^2)O(n2) — bubble, insertion, selection. Fine for very small arrays.
  • O(nlog⁡n)O(n \log n)O(nlogn) average — quicksort, the workhorse.
  • O(nlog⁡n)O(n \log n)O(nlogn) worst case — merge sort, heap sort. Quicksort’s worst case is O(n2)O(n^2)O(n2) unless you randomize the pivot.

Constants matter more than the asymptotic class for the inputs you actually see. Quicksort beats merge sort on most real inputs despite the same big-O.
